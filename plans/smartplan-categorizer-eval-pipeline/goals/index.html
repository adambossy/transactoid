Why this exists

Goals & success criteria

What the pipeline must produce: a continuous quality signal that guides ongoing improvement, not a one-time ship gate.

Requirements

  1. The owner can see, each day, how often the new categorizer matched a human judgment on brand-new transactions.
  2. The owner can compare the new categorizer head-to-head against the current one and see which the human kept.
  3. The owner can watch the accuracy trend over weeks and tell whether a change actually helped.
  4. The owner has a continuous signal of categorizer quality that guides ongoing improvement, with no fixed finish line.

Outcomes we want

  • A daily number: how often the new agent agrees with a human on fresh transactions.
  • A side-by-side: agent vs legacy, so we can see which one the human keeps.
  • A trend over weeks, annotated where the categorizer version changes, so improvement is attributable to a specific prompt/taxonomy change rather than to easier data.

A continuous quality signal (no fixed "ready" state)

Decision (2026-06-27): there is no numeric ship gate. The agent is improved gradually and indefinitely; the eval is a permanent quality monitor, not a one-time acceptance test, and it keeps running even after the agent replaces legacy in prod. The numbers below are how we watch quality, not a bar that flips a switch:

  • Agent exact-match accuracy on agent-decided rows (fast path excluded), tracked over a rolling window.
  • Agent win-rate vs legacy (human prefers agent vs prefers legacy).
  • Parent-level regressions vs legacy on the same cohort — the signal that a change made things worse.

What matters is the direction of the trend across categorizer versions, not any single threshold.

Non-goals

  • Not building a labeling product — corrections happen in the existing chat for the MVP.
  • Not auto-promoting the agent to prod; the decision to ship stays manual and human-read.
  • Not evaluating legacy in isolation beyond the baseline it provides for comparison.
