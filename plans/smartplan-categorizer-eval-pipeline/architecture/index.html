How it fits together

Architecture overview

The transient-branch + R2-fixture design and the unattended-then-async sequence that runs every 12 hours.

Requirements

  1. A maintainer can read one diagram and understand exactly what runs unattended versus what waits for a human.
  2. The new categorizer is exercised on the same history the production run saw, so the comparison is fair.
  3. The evaluation never alters production data, even though the new categorizer writes as it runs.

Three separated concerns

  • System under test — the production 12-hour sync, categorizing with the legacy method. Unchanged.
  • Snapshot — a Neon branch off prod on each 12-hour sync (Postgres is first-class). The agent replays here. Transient: dumped to a SQLite fixture in R2 and deleted at the end of the run, with no human gate; R2 holds it for perpetual backtesting.
  • Measurement — a durable prod-resident eval record (agent decision, legacy baseline, metrics, R2 fixture reference). Right/wrong is read later from your natural corrections. Metrics are a read over this.

Keeping these apart is what makes a stateful categorizer A/B-able: the agent's writes never pollute prod, and it reads the same history the legacy run saw.

The per-run sequence (every 12h)

sequenceDiagram participant Prod as Prod DB participant Cron as 12h sync and eval participant Branch as Neon branch participant Store as Eval store (prod) participant You as You (chat) participant R2 as R2 (fixtures) Note over Prod: 12h sync categorizes new rows, LEGACY baseline Cron->>Branch: 1. branch off prod, fail closed on error Cron->>Branch: 2. replay NEW agent over this run cohort Branch-->>Cron: agent decision, reasoning, confidence, trace link Cron->>Store: 3. write durable records (legacy vs agent, version stamp) Cron->>R2: 4. dump branch to SQLite fixture Cron->>Branch: 5. delete branch Cron-->>You: 6. email report if any disagreements You->>Store: 7. later: recategorize wrong rows (read as corrections) Note over Store,R2: metrics computed lazily, backtests hydrate SQLite from R2

Steps 1–6 all run unattended in the single 12-hour run — the branch is created, used, and deleted without waiting on anyone. Step 7 is asynchronous: you review the emailed report and improve the categorizer in code (or recategorize a wrong row), and the eval reads those corrections later. The agent replays under the same advisory-lock boundary the prod sweep uses, so it cannot race production.
