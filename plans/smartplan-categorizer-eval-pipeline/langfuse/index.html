Langfuse role

Trace deep-links for debugging only — Scores/Datasets/Experiments don't fit OTEL push-only ingestion.

What Langfuse does for us

Tracing is already wired: penny.observability exports OTEL spans to Langfuse, and the harness emits a per-run trace. The eval artifact includes a best-effort trace deep-link per transaction (the existing _current_trace_link) so a surprising decision can be opened and its model turns, tool calls, and web searches inspected. That is the whole role: debugging.

Why not Scores / Datasets / Experiments

A P0 review finding: Penny's tracing is OTEL push-only (BatchSpanProcessor + OTLPSpanExporter); there is no Langfuse SDK client. Scores, Datasets, and Experiments are SDK/REST features keyed by Langfuse's own trace id — and the OTEL trace id we hold is not guaranteed to be the id Langfuse stores against the ingested trace. So "attach a Score by trace id" is not reliably addressable, and Datasets/Experiments are an entire integration that does not exist. Relying on them would put the load-bearing part of the eval on a feature we haven't wired.

Hence: verdicts and the golden set live in our prod eval store, and the trend is our own page.

Out of scope: a Langfuse SDK spike

If, later, we want native Scores/Datasets/Experiments (e.g. to run backtests in the Langfuse UI), that is a separate spike: add the SDK client, confirm trace-id addressability under OTLP ingestion, then mirror verdicts as Scores. Not part of this plan.
