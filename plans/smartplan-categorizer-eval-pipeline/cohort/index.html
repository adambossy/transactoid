Daily Categorizer Evaluation Pipeline /  Cohort identification

What gets evaluated

Cohort identification

This run's cohort is rows ingested since the last run, addressed by created_at — not posted_at.

Requirements

  1. Every transaction that newly arrived today is evaluated, even if its bank date is older.
  2. A transaction is never double-counted or missed because of how its date is interpreted.
  3. On a day with no new transactions, the pipeline records a clean gap rather than an error.

What "this sync's cohort" means

The eval runs on every 12-hour sync, so the cohort is the transactions ingested by this sync — the rows that arrived since the previous run — addressed by created_at, not posted_at. Plaid back-dates pending→posted transitions and late arrivals, so a posted_at filter both misses transactions that posted earlier but synced now and is empty on quiet windows. This was a P0 review finding.

Measured prod volume (2026-06-21): ~20–23 transactions/day (last 30–90 days) — about ~10 per 12-hour run, a reviewable batch. The 12-hour cadence was chosen for exactly this: large enough that runs are rarely empty, small enough to review in one sitting. After per-descriptor dedup the number of agent runs per cohort is smaller still.

The selection query (one new facade method)

Add a single method to facade.py; do not invent a cohort table.

  • Primary: derived_transactions.created_at > the previous run's high-water mark.
  • Equivalent: transaction ids whose earliest transaction_category_events.created_at falls in-window (the events log is already timestamped and append-only).

Note the existing get_uncategorized_derived_ids (category_id IS NULL) does not work here: by branch time the legacy sync has already set category_id, so the cohort would be empty.

Edge cases

  • Empty cohort (uncommon at the 12-hour cadence): no branch is cut; record an eval_runs row with status = skipped_empty.
  • Window boundaries: define the cohort as "ingested since the previous successful eval run" (a high-water mark on created_at) rather than a fixed clock window, so a late or skipped sync never drops transactions. created_at is a UTC server timestamp.
  • Re-categorized-before-eval: if a row was already manually recategorized before the eval ran, treat its current category as the baseline and flag it (rare).
