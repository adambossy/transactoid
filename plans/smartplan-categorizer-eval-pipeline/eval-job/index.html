The unattended half

The eval job (cron)

On each 12-hour sync, all in one unattended run: branch, replay, write durably, dump to R2, delete the branch, email if anything disagreed — with fail-closed and no-op guards.

Requirements

  1. The twice-daily job runs without supervision and produces a review page for the owner.
  2. If anything about the snapshot looks wrong, the job refuses to record results rather than recording bad ones.
  3. Re-running an eval can never silently inflate or corrupt its numbers.

What runs on each 12-hour sync

  1. Resolve this run's cohort against prod (read-only) via the new facade method (ingested since the last successful eval run).
  2. If empty or the legacy baseline is incomplete → write a skipped_* run row, no branch, exit 0.
  3. Create the Neon branch (see Branch lifecycle & R2). On any branch/connection error → fail closed: write nothing, exit non-zero.
  4. Replay the agent on the branch, capture decisions, write durable eval_* records to prod.
  5. Dump the branch's reachable tables to a SQLite fixture, upload to R2, store its r2_fixture_url on the run, mark status = completed, and delete the branch — all in this same unattended run.
  6. If the cohort has any legacy≠agent disagreements, email the report; a fully-agreed run sends nothing. You review later and improve the categorizer in code (or recategorize a wrong row); the eval reads those corrections lazily. No staging step.

Replay & capture

Reuse categorize_one and the harness's _run_agent_traced wrapper so the agent's per-turn spans are traced exactly as in production. For each cohort row capture, in-process:

  • agent_key (= the persisted to_category_key),
  • agent_reasoning, tools consulted, web-search summary,
  • agent_confidence — only available from the tool call at run time; it is never written to the DB, so it must be captured here,
  • method_at_eval_time — whether the fast path fired.

Write durable records to prod

As the agent runs, write one eval_items row per cohort transaction to the prod eval store, plus the eval_runs header with the version stamp — so the measurement survives the branch deletion. legacy_key comes from the agent event's from_category_key. The r2_fixture_url is written at the end of the same run, right after the dump.

Idempotency & re-runs

Each run maps to one timestamped eval_run_id; the cohort high-water mark guarantees a transaction is evaluated by exactly one run. Re-running a given run is idempotent on that id (delete-and-reinsert its items), never appended. The branch is gone as soon as the run completes, so any later re-run is a backtest that hydrates SQLite from r2_fixture_url (see Branch lifecycle & R2).
