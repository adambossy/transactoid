Daily Categorizer Evaluation Pipeline /  Human review loop

The async half

Human review loop

Get the report by email, improve the categorizer in code or recategorize a wrong row — no stamp or staging step.

Requirements

  1. The owner can review a day's results whenever convenient, without the pipeline waiting on them.
  2. A verdict always refers to exactly what the agent decided that day, even if the data changed since.
  3. Correcting a category in review uses the same simple chat action the owner already knows.

The review flow (MVP)

Each 12-hour run writes its records, dumps to R2, and deletes its branch on its own. If the run has any legacy≠agent disagreements, its HTML artifact — every cohort transaction with legacy vs agent, the agent's reasoning, tools consulted, web-search summary, confidence, and a Langfuse trace deep-link — is emailed to the owner. There is no staging or stamp step: you read the email whenever, and improve the categorizer by editing code (prompt / taxonomy / rules) and/or recategorizing a wrong row in chat via the existing recategorize_transaction tool.

Fully-agreed runs are silent. When legacy and agent agree on every row, there is nothing to review, so no email is sent — the run still records its metrics and R2 fixture. You only ever get email for runs that actually need your eyes.

How right/wrong is recorded

No explicit verdict is captured at review time. Right/wrong is derived from your natural corrections: when you recategorize a cohort transaction (a manual recat event after the run's run_at), the eval reads that as "the agent was wrong here" — corrected, with your new key. A row you never recategorize counts as correct once the settling window has passed (you saw the emailed report and let it stand).

The verdict therefore binds to the durable (eval_run_id, transaction_id) record, computed lazily from the events log — not to the live prod category, which may move later. Your chat correction updates prod for your real data as usual; the eval simply reads it.

Honesty note. Without a confirm step, "uncorrected" can't be told apart from "not-yet-reviewed" until the settling window elapses. Emailing every disagreement run is the mitigation — it ensures the runs that need eyes actually get them.

Deferred: a dedicated correction UI

A purpose-built HTML review-and-correct screen (one-click confirm/correct) is explicitly out of scope for now. Chat-based correction is the MVP; the HTML artifact is read-only. Revisit once the daily volume makes chat tedious.
