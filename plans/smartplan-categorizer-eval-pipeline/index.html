Penny · categorizer

Daily Categorizer Evaluation Pipeline

A twice-daily (every 12h), human-adjudicated A/B of the new categorizer agent against the legacy incumbent, tracked as a versioned accuracy trend aggregated by day.

The goal

Penny categorizes bank transactions against a two-level taxonomy. Production today uses the legacy single-shot categorizer. A new per-transaction agent (with history tools and web search) is the candidate replacement. Before it ships, we need evidence — measured on real, fresh data — that it is at least as good as the incumbent, and a trend line that shows it improving as we tune its prompt and taxonomy.

This plan builds an evaluation that runs on every 12-hour sync: after each sync, replay the new agent on the just-synced transactions, A/B each decision against the legacy categorization, present the disagreements to a human for a thumbs-up-or-correct verdict, and append the result to a versioned time series aggregated by day. At Penny's volume that is a reviewable batch of ~10 transactions per run — see the data note.

The strategy

Three concerns are kept strictly separate. The system under test is the production 12-hour sync, which categorizes new rows with the legacy method — left untouched. The snapshot is a Neon branch cut off prod on each sync; the agent replays there (Neon/Postgres is Penny's first-class database) so its writes never touch prod and it reads the same frozen history the legacy categorizer saw. The branch is transient: at the end of the run the DB is dumped to R2 as a SQLite fixture and the branch is deleted — nothing waits on a human. The measurement is a durable, prod-resident eval record — the agent's decision, the legacy baseline, metrics, and a reference to the R2 fixture; right/wrong is read later from the corrections you make naturally, and metrics are computed lazily.

The branch is the keystone of the live run: it makes the legacy-vs-agent write conflict a non-issue (they live on different snapshots) and the append-only events log records both decisions as a single from→to row. The R2 fixture is the keystone of perpetuity: the eval lives forever as a cheap object a future categorizer version can be backtested against. See Architecture overview.

Key tradeoffs

  • Re-run, don't just read. We replay the new agent rather than grading a single stored decision, because we are A/B-ing two different categorizers on the same input.
  • Langfuse is debugging-only. Penny emits OTEL spans push-only; there is no SDK client, so Scores/Datasets/Experiments are out. Verdicts and the trend live in a local prod-resident store. See Langfuse role.
  • Async, no staging step. Cron cannot block on review. The durable record is written when the agent runs; you review the emailed report and improve the categorizer in code (or recategorize a wrong row), and the eval reads those corrections later — there is no "stamp" or approval button.
  • Neon for the live run, R2 forever. Each run dumps the branch to R2 (a few MB) and deletes it at the end, so branches stay short-lived and inside the included allowance (~$0). Perpetual retention lives in R2 at pennies; backtests hydrate a throwaway SQLite from the R2 fixture. See Branch lifecycle & R2.
  • This change deletes the "blind" eval mechanism. The A/B + human adjudication replaces it. See Blind removal.
