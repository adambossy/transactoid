What we store

Data model & eval store

Two durable prod tables for the verdict and the version stamp — everything else is a join over the append-only events log.

Requirements

  1. The owner's verdict on a transaction survives even after that run's branch is deleted.
  2. Each run's measurement records exactly which version of the categorizer produced it, and where its fixture lives.
  3. The same fact is never stored twice, so the comparison can't drift out of sync with itself.

The eval store (prod-resident, own namespace)

Durable because the branch is deleted at the end of each run and the snapshot then lives only in R2. The store is in the prod DB but a separate concern from the product tables. Two tables:

eval_runs — one row per 12-hour run: eval_run_id (PK), run_at, branch_name, cohort_size, status (completed / skipped_empty / skipped_incomplete_baseline), r2_fixture_url (written at the end of the run), and the version stamp (see below). No review-state column — there is no staging step.

eval_items — one row per cohort transaction: (eval_run_id, transaction_id) PK, merchant_descriptor, legacy_key, agent_key, agent_reasoning, agent_confidence, method_at_eval_time (fast_path | agent), trace_link. The verdict is not stored at write time; it is derived later from your corrections: a manual recat of this transaction after run_at means the agent was wrong (corrected, with the new key), otherwise it counts as correct once a settling window has passed.

Reuse the events log; don't duplicate the baseline

On the branch, the agent's submit_categorization overwrites the row in place and appends a transaction_category_events row whose from_category_key is the legacy key and to_category_key is the agent key. Legacy-vs-agent is therefore one event row — no separate baseline table (explicitly rejected). The eval store's only genuinely new facts are agent_confidence (never persisted to the DB; captured in-process at run time), method_at_eval_time, the version stamp, and the run's r2_fixture_url. The verdict is not stored — it is derived from the corrections in the events log.

The version stamp

Every eval_runs row records what defined the agent's decision space that day, so the trend can attribute change to a cause:

  • model id (e.g. gemini-3.5-flash)
  • categorizer prompt version (categorize-transaction-agent/<n>)
  • agent-harness git SHA
  • taxonomy version — the key set is the grading target
  • merchant-rules version — also shapes decisions

Taxonomy and merchant-rules versions are easy to forget and were a review finding: omit them and a taxonomy rename looks like a model regression.
