Rollout & verification

Five dependency-ordered commits on this feature branch, each individually green; a safe additive migration; and a two-member manual pass before merge.

Requirements

  1. Every commit leaves the app working — a mid-stack revert never strands the UI against a missing API.
  2. The production database change is additive, reversible, and applied by the normal deploy pipeline, not by hand.
  3. Before merge, the feature has been seen working in a real browser with two signed-in household members.

Commit order

  1. feat(api): record message sender — model column, migration 025, store stamping, hydration field, tests. Invisible to users; deployable alone. (details)
  2. feat(api): household members endpoint — Clerk adapter profile fetch, GET /api/household/members, TTL cache, tests. (details)
  3. feat(api): expose session_mode on conversation list — one-key response change + test. Could fold into commit 2 if it feels too small alone. (details)
  4. feat(ui): avatar primitives + members hook + sidebar stack — Avatar/AvatarStack, useHouseholdMembers, drawer entry restyle. First visible change. (details)
  5. feat(ui): green bubbles for the other member — OtherUserMessage, the render branch, sender map. Includes the REQUIREMENTS.txt product-section update (below). (details)

Backend-first ordering means the frontend never ships against an API that is not already live; each frontend commit degrades gracefully if its data is missing.

Automated verification

Per unit of work, the standard gate from backend/: uv run ruff check ., uv run ruff format --check ., uv run pytest -q. Additionally for this feature:

  • Postgres suite (POSTGRES_TEST_URL=… uv run pytest -q -m postgres) after the migration commit — it exercises RLS paths the SQLite run skips, and migration 025 touches an RLS-governed table.
  • Schema drift test (tests/test_schema_drift.py) confirms models and the migration chain still agree.
  • Frontend: the repo has no frontend test gate; rely on TypeScript (npm run build) and the manual pass. The tokens.ts drift test applies only if a new color token is minted (the plan avoids one).

REQUIREMENTS.txt — same change, not after

This is a user-facing behaviour change, so the Product section of REQUIREMENTS.txt must gain the two behaviours in the same branch (reviewers treat it as scope-of-record):

  • Shared (joint) conversations are visually identified in the chat history by the participating members' profile pictures.
  • Within a shared conversation, messages are attributed to their author; another member's messages are visually distinct from the viewer's own.

No Technical-invariant change: avatars are deliberately not persisted (worth a sentence in the PR description rather than a new invariant).

Manual two-member pass

The dev auth stub pins a single principal per process, so a genuine two-member test needs two Clerk users in one household against the Neon penny-test branch (the invite flow creates the second member). Checklist, using two browser profiles:

  1. Member A starts a joint conversation and sends a message; member B's drawer shows the entry with the two-avatar stack; A's individual conversations show none.
  2. B opens the conversation: A's message is green, left-aligned, with A's Google picture; B replies and their own message is grey/right.
  3. A reloads: B's reply is now green-with-avatar on A's screen; A's own messages unchanged.
  4. Open a pre-feature joint conversation: old messages render as today (no green, no avatar) — the legacy-rows contract.
  5. Temporarily break the members fetch (bad token or blocked route): chat still works, drawer shows no stacks, no console errors beyond the failed fetch.

Quicker single-member smoke tests along the way (avatar fallback, sidebar layout) work in dev mode with Clerk absent, since the members endpoint degrades to initials.
