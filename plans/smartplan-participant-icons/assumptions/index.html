Assumptions & open questions

What was asked, what was assumed by default, and what remains genuinely unresolved.

What I asked you

QuestionYour answerWhere it lands
How should Penny obtain profile pictures? "Just keep using the Clerk icon directly, as we do now — no complicated syncing or persistence." Shapes the whole avatar design: live proxy via one endpoint, no schema change, no webhooks. Members & avatars API
What makes a conversation "multi-user"? Joint mode + household of 2+ members (structural rule, not actual senders). Conversations API, History sidebar
Layout of the other member's bubbles? Left-aligned green with avatar (iMessage-style); own messages unchanged on the right. Green bubbles

One nuance on the first answer: "as we do now" — the app currently shows a Clerk avatar only for the signed-in user (Clerk's own account button). Clerk's browser SDK cannot see other members, so honouring "no syncing, no persistence" still requires the thin backend proxy that fetches the other member's image URL from Clerk's server API on demand. That is the minimal faithful reading of the answer; nothing is stored anywhere.

What I assumed without asking

AssumptionAlternativeWhy this defaultAffects
A ~5-minute in-process cache on Clerk profile fetches counts as "not complicated". No cache at all — one Clerk call per member per drawer open. It is a dict with timestamps, not infrastructure; without it the sidebar hammers Clerk on every open. Members API → caching
Legacy (pre-feature) messages render as the viewer's own, exactly as today. Render unattributed messages in a third neutral style. Backfill is impossible; preserving today's look means the feature cannot make old conversations worse. Sender attribution, Green bubbles
The green is the existing steel token (#3C7A72) with paper-white text. Mint a new soft-green token (theme.css + tokens.ts). Reuses the palette's only green, avoids drift-test churn; easy to swap after seeing it in situ. Green bubbles → style
No participants junction table — a joint conversation's participants are simply the household's members. Model per-conversation membership explicitly. Matches how access control already works (joint = household-visible); a junction table would model a concept the product does not have yet. Backend → boundaries
The stack shows at most two avatars, other member in front. Show up to three, or a "+N" overflow chip. Households are two-person today; the "+N" chip is a five-line follow-up when a third member becomes real. Avatar primitives, Sidebar
Sender ids ride a side map in the chat screen, not inside the AI SDK message objects. Attach metadata to the SDK's message state. Keeps the SDK's typing/state management untouched; hydration already flows through Penny-owned code. Green bubbles → sender resolution

Open questions

  • Should the other member's name appear near their bubble? With only two people an avatar may suffice; group-style sender labels become useful if households grow. Affects Green bubbles only — a one-line addition either way. Suggest deciding by eye during the manual pass.
  • Does the other member's message need to appear without a manual reload? There is no live push between members today; attribution rides the reload path. If "live" shared conversations are wanted later, that is a separate (websocket/poll) project — noted so nobody mistakes this plan for delivering it. Affects Green bubbles → streaming.
  • Clerk rate limits at scale. Irrelevant at current size; if Penny ever has many households per instance, revisit the rejected webhook-sync design. Affects Members API → caching.
