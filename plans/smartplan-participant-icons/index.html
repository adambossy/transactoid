Penny · chat UI

Participant icons & multi-user chat

Make shared household conversations legible: overlapping member avatars on multi-user entries in the history sidebar, and green, avatar-tagged bubbles for messages the other household member sent.

Requirements

  1. A household member scanning the chat history can tell at a glance which conversations are shared with their partner, because those entries carry two small overlapping profile pictures on the right.
  2. Inside a shared conversation, messages written by the other member look unmistakably different from the reader's own: green, on the left, with the sender's profile picture beside them.
  3. Profile pictures are the ones people already have on their Google account — nobody uploads or configures anything in Penny.
  4. Nothing changes for a solo household or an individual conversation; the UI looks exactly as it does today.

The goal

Penny already supports shared conversations — a joint conversation is visible to every member of the household — but the UI hides that fact. The history sidebar renders every conversation identically, and inside a conversation every human message renders as "yours": right-aligned, grey, anonymous. In a two-person household this is actively confusing: you can open a joint conversation and read messages you never wrote, presented as if you did.

This project makes shared conversations legible in two places: the sidebar (who is in this conversation?) and the transcript (who said this?).

Strategy

Exploration of the codebase found two genuine gaps behind the two visible features, plus one deliberate non-goal:

  1. Messages carry no sender identity. The message table records only user vs assistant. The backbone of the work is a new nullable sender column, stamped at write time and surfaced when a conversation is loaded. See Sender attribution.
  2. Penny stores no profile pictures — and will not start. Per the decision taken during planning, avatars come straight from Clerk (which mirrors the Google account picture) with no syncing or persistence. Your own picture is already available in the browser; the other member's is fetched through one small backend endpoint that proxies Clerk. See Members & avatars API.
  3. No changes to the shared UI library. The green bubble and avatar are rendered by Penny's own frontend around the existing message component, not by forking @adambossy/agent-ui. See Green bubbles.

Shape of the work

The work splits cleanly along the API boundary, then lands in five dependency-ordered commits (detailed in Rollout):

  • Backend — three small changes: the sender column and its migration, a household-members endpoint that returns Clerk image URLs live, and one added field on the conversation-list response.
  • Frontend — an Avatar/AvatarStack primitive pair, a members hook, the sidebar entry restyle, and the green-bubble render branch in the chat screen.

Everything backend lands first and is invisible on its own; the frontend then consumes it. No step is destructive and every schema change is additive and nullable.

Big tradeoffs

  • Live Clerk fetch vs. stored avatars. Chosen: live fetch, no persistence. It keeps the schema untouched and pictures always current, at the cost of a runtime dependency on Clerk for avatar display. A short in-process cache keeps the cost to roughly one Clerk call per member every few minutes. If Clerk is briefly unreachable, avatars degrade to initials — the feature, not the app, degrades.
  • Sidebar rule is structural, not literal. A conversation shows the avatar stack when it is joint and the household has two or more members — even before the second person has actually typed. Chosen deliberately: it is cheap (no message scan) and it advertises "this is a shared space", which is what the joint mode means.
  • Old messages stay anonymous. Messages written before this ships have no sender recorded and cannot be backfilled. They keep today's rendering (grey, right-aligned). Only messages written after the backend lands get attribution.
