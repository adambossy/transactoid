Green bubbles

Feature 2: in ChatScreen, messages the other household member sent render as left-aligned green bubbles with their avatar — the iMessage-style layout chosen during planning. Everything else renders exactly as today.

Requirements

  1. Reading a shared conversation, the reader can attribute every recent human message to its author without thinking: mine are grey on the right, theirs are green on the left with their face.
  2. Penny's own replies are visually unchanged.
  3. Messages from before this feature existed keep their current appearance rather than guessing an author.

The render branch

The message map (frontend/src/ChatScreen.tsx:275–283) wraps each message in <div data-role=…> and delegates to agent-ui's Message. Insert one branch:

flowchart TD m[message] --> u{role is user} u -- no --> lib[agent-ui Message, unchanged] u -- yes --> s{sender known and not me} s -- no --> lib s -- yes --> other[OtherUserMessage, green bubble plus avatar]

OtherUserMessage is a small local component in frontend/src: a flex row of Avatar size="sm" and a bubble div rendering the message's text parts. User messages are plain text (only assistant messages carry tool/reasoning parts), so reproducing the bubble locally forfeits nothing from the library.

Bubble style

Mirror the library's own user bubble (rounded-2xl px-4 py-2.5 max-w-[78%]) but left-aligned and green. For "green" in Penny's warm palette, reuse the existing steel token (#3C7A72) — bg-steel text-paper — rather than minting a new color: it is the palette's green, it clears WCAG AA against paper-family text at bubble sizes, and reusing it means no theme.css/tokens.ts drift-test churn. The avatar sits bottom-left of the bubble (items-end), 4px gap.

Penny is forced-light (the dark: variant is retargeted to a never-set class), so one value suffices — no dark-mode variant needed. If bg-steel reads too saturated in situ, the fallback is a new soft token (e.g. --color-sage) added to packages/ui/src/theme.css and mirrored in tokens.ts, which the drift test guards.

Resolving "mine vs theirs"

Two lookups, both already planned:

  • Sender per message: when ChatScreen hydrates a session it receives senderUserId per message (see Sender attribution). Keep a sendersById: Map<messageId, userId> in component state, populated from the hydration payload — deliberately outside the AI SDK's message objects, so useChat's state handling is untouched.
  • The viewer: me.user_id from useHouseholdMembers.

Decision rule: green iff the sender is known and is not me. A missing sender (legacy rows, or members still loading) falls through to today's rendering — wrong-styling-by-guess is worse than the status quo.

Streaming and live turns

Messages the viewer sends live never enter sendersById (they are not in the hydration payload), so they take the "mine" path automatically — correct by construction, since the sender of a live turn is always the viewer. There is no live push between household members today (no websocket); the other member's messages appear when the conversation is reloaded/hydrated, which is exactly the path that carries attribution. Nothing extra to build.

Verification: a quick browser pass with two members (see Rollout → manual test) covering hydrated other-user messages, own live sends, and a legacy conversation.
