Sender attribution

Record which household member wrote each human message, and hand that identity to the client when a conversation is loaded. This is the single schema change in the project and the backbone of the green-bubble feature.

Requirements

  1. From the moment this ships, every message a person sends is stored with their identity attached.
  2. When anyone in the household reopens a conversation, each human message says who wrote it.
  3. Messages written before this shipped keep working exactly as today — nothing breaks because older rows have no sender.

Schema change

Add one nullable column to ConversationMessage in backend/penny/api/persistence/models.py (table web.conversation_messages, currently lines 87–139):

  • sender_user_id — UUID, nullable, no foreign key. It identifies the household member who wrote a user-role message; it stays NULL for assistant messages and for all rows written before the change.
Why nullable, and why no foreign key

Nullable because assistant rows and legacy rows legitimately have no sender — making absence a normal state defines the backfill problem out of existence. No foreign key because the web schema already references household_id/owner_user_id as bare UUIDs with no cross-schema FK (see the note on Conversation); a FK to the finance-side users table would pierce the schema boundary the web store deliberately keeps.

Stamping at write time

store.append_user_message (backend/penny/api/persistence/store.py:230–257) currently writes role="user" and discards the caller's identity, even though the chat route has it in hand. Change: the route (backend/penny/api/main.py:521–523) passes the sender explicitly, and the store writes it.

One trap: use the real ctx.user_id, not ctx.effective_user_id. The tenancy context's effective_user_id deliberately collapses to the nil UUID in joint sessions (penny/tenancy/context.py:23–26) — exactly the sessions where attribution matters most. Attribution is about the human, not the tenancy principal.

Assistant writes (upsert_assistant_message) are untouched — Penny's messages need no sender.

Surfacing on load

conversation_to_ui (backend/penny/api/hydration.py:20–37) shapes each stored row into the AI SDK message form {id, role, parts} for GET /api/sessions/{id}. Add senderUserId (string or null) alongside those three fields on user-role messages.

The field rides the hydration payload only — the live streaming path (POST /api/chat) does not need it, because the sender of a live turn is always the viewer themself (see Green bubbles → streaming).

Migration

Two authorities, per the repo's schema rules:

  • SQLite (dev/test): the ORM model is the authority; create_all via create_web_schema picks the column up automatically. Nothing to do beyond the model edit.
  • Postgres (prod): add alembic revision 025_add_message_sender (down-revision 024_tenant_guc_wrapper) following the established web-table pattern of revisions 019/023: return early when the dialect is not postgresql, then ALTER TABLE web.conversation_messages ADD COLUMN IF NOT EXISTS sender_user_id uuid. Downgrade drops the column.

Additive and nullable means the migration is safe to run against live data with no backfill, no lock concerns beyond an instant catalog change, and a trivial rollback.

Legacy rows and tests

Rows written before the change hydrate with senderUserId: null. The frontend's contract (see Green bubbles) is: a null sender renders as the viewer's own message — i.e. exactly today's behaviour. No backfill is attempted; attribution begins at deploy time.

Tests to add alongside the existing store/hydration suites:

  • append_user_message stamps the passed sender; assistant upserts leave it null.
  • A joint-session write stamps the real user id, not the nil UUID.
  • conversation_to_ui emits senderUserId on user rows and null on legacy/assistant rows.
  • Idempotent re-append (same ai_sdk_message_id) does not clobber an existing sender.
