Backend

Three additive changes to the website/app domain — a sender column on messages, a members endpoint that proxies Clerk avatars, and one new field on the conversation list. No finance-side or agent-domain code is touched.

Requirements

  1. Every human message written from now on records which household member wrote it, and that identity travels with the message when a conversation is reopened.
  2. The app can list the household's members with their current Google profile picture, without Penny ever storing a copy of the picture.
  3. The conversation list tells the client enough to decide which entries are shared spaces.
  4. All of this stays inside the website domain: no agent tool, skill, or finance-facade code changes, and app data stays out of the agent's reach.

Data flow

sequenceDiagram participant B as Browser participant A as Penny API participant C as Clerk B->>A: GET /api/household/members A->>C: fetch profile per member, short cache C-->>A: image URL and name A-->>B: members with is_you flag B->>A: GET /api/sessions/id A-->>B: messages, each with senderUserId

The browser joins the two streams itself: a message's senderUserId is looked up in the members list to find the avatar and to decide "mine vs theirs". The backend never assembles a per-message avatar.

Domain boundaries respected

  • Website domain only. The sender column lives on the web-schema message table, out of the agent's run_sql reach. The members endpoint sits beside the existing household routes in signup_routes.py.
  • The Clerk adapter is the single seam to Clerk. The new profile fetch extends penny/adapters/clerk.py, which already owns invitations and the one-shot identity fetch — no route calls Clerk's REST API directly.
  • Membership is not modelled twice. The participant set of a joint conversation is simply "the members of its household" — the existing users WHERE household_id = … query. No participants junction table is introduced; see the assumptions page for why.
