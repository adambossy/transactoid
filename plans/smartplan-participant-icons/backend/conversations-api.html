Conversations API

The smallest change in the project: the conversation list starts saying which entries are joint, so the sidebar can decide where the avatar stack belongs.

Requirements

  1. The chat history the app receives says, per conversation, whether it is a personal or a shared one.
  2. The list stays as fast and small as it is today — no per-conversation lookups are added.

Response change

GET /api/conversations (backend/penny/api/main.py:402–422) today returns {id, title, updated_at} per conversation, even though the store already hands the route full Conversation rows carrying session_mode. The change is one added key in the response dict:

{ "id": "…", "title": "Groceries plan", "updated_at": "…",
  "session_mode": "joint" }

No store change, no query change, purely additive — existing clients ignore the extra key.

The multi-user rule lives in the client

Per the decision taken during planning, a conversation counts as multi-user when it is joint and the household has two or more members. The backend supplies the two facts separately — session_mode here, the member list from the members endpoint — and the sidebar combines them:

isMultiUser = conversation.session_mode === "joint" && members.length >= 2

Two things were considered and rejected:

  • A per-conversation member count from the server — redundant: membership is household-wide, one members fetch serves every entry in the list.
  • "Two or more distinct actual senders" — more literal but needs a message-table aggregate per conversation and shows nothing until the second person speaks; the structural rule advertises the shared space immediately.

Test: the route test asserting the response shape gains the session_mode key for both an individual and a joint conversation.
