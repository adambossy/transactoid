Members & avatars API

One new endpoint, GET /api/household/members, that lists the household's members with their current Clerk (Google) profile picture — fetched live, never persisted, per the decision taken during planning.

Requirements

  1. Anyone signed in can see who is in their household, with each person's current Google profile picture and a usable display name.
  2. Penny never stores a copy of anyone's picture; if someone changes their Google photo, Penny shows the new one within minutes with no sync job.
  3. If the picture service is unreachable, the app still works — members simply show initials instead of photos.
  4. Nobody can list members of a household they don't belong to.

Endpoint shape

GET /api/household/members, added to backend/penny/api/signup_routes.py beside the existing household routes (/api/me, /api/invites). Scoped by the request context's household_id like everything else there. Response:

{ "members": [
  { "user_id": "…", "email": "ada@…", "display_name": "Ada",
    "image_url": "https://img.clerk.com/…", "is_you": true },
  { "user_id": "…", "email": "sam@…", "display_name": "Sam",
    "image_url": null, "is_you": false }
] }
  • Members are the finance-side users rows for the household with a linked login (external_auth_id IS NOT NULL) — pending invitees are rows too, but they have no Clerk account and no picture, so they are excluded. The existing invites screen already covers them.
  • display_name is Clerk's first name when present, else the email local-part — a server-side decision so every client renders names the same way.
  • is_you lets the client find the viewer without a second call to /api/me.
  • image_url is null when Clerk has no image or the fetch failed; the client falls back to initials.

Clerk adapter extension

backend/penny/adapters/clerk.py is the single seam to Clerk (invites + fetch_user_identity, lines 109–150, which already calls GET /v1/users/{id} and reads only the email). Add a sibling fetch_user_profile(external_auth_id) reusing the same HTTP plumbing, returning {image_url, first_name, last_name} from the same Clerk payload that fetch_user_identity already receives and discards.

Clerk's image_url values are publicly loadable img.clerk.com URLs, so returning the URL to the browser is sufficient — no image proxying, no signed URLs, no bytes through Penny.

A failed or slow Clerk call for one member must not fail the endpoint: catch per-member, return image_url: null for that member, log at warning. This is "define errors out of existence" — a missing picture is a normal state the client already handles, while a genuinely broken route (bad auth, DB down) still raises.

Caching — the one knob

The sidebar opens often; households are tiny (one to a few members). Without any cache, every drawer open costs one Clerk API call per member. Proposal: a module-level in-process TTL cache keyed by external_auth_id, roughly 5 minutes — a dict and a timestamp, not a cache library. That bounds Clerk traffic to about one call per member per app instance per 5 minutes while keeping "no complicated syncing" intact: nothing is persisted, nothing survives a restart, nothing needs invalidation plumbing.

This is deliberately the smallest thing that works. If it ever matters more (many instances, rate limits), the escape hatch is the webhook-sync design that was considered and rejected — recorded on the assumptions page.

Edge cases & tests

  • Dev mode (env-pinned principal, no Clerk): the endpoint still returns the member list from the DB with image_url: null throughout. The UI shows initials — the feature is exercisable locally without Clerk keys.
  • Solo household: returns one member; the frontend simply never shows a stack (see multi-user rule).
  • Tenancy: the query filters on ctx.household_id; a test asserts a member of household A never appears for a caller in household B.

Tests: route test with the Clerk adapter mocked (happy path, Clerk failure → null image, pending invitee excluded, cross-household isolation); adapter test for the new fetch parsing image_url/first_name from a canned Clerk payload.
