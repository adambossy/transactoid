Persisting Chat Conversations & Messages  /  Data model

Section 1

Data model

Two tables on the website's own Base: conversations and conversation_messages. Each message carries its ordered parts as a JSON array shaped to match the AI SDK frames the bridge emits, so hydration is near-passthrough.

Requirements

  1. A stored message can be read back and rendered without re-deriving anything from the model's transcript.
  2. Re-sending or reconnecting with the same message never creates a duplicate row.
  3. Messages always replay in the exact order they were said.

Why a JSON parts column

  • Typed columns (one per part attribute) — rejected: parts are a heterogeneous union (text/reasoning/tool-call/tool-output/error/step markers) with wildly different shapes; forces a sparse mega-row.
  • Separate polymorphic parts table — defensible and textbook-normalized, but the AI SDK UIMessage is itself {id, role, parts: [...]}; we never query across parts; ordering is just array index (no ord column + join needed); one row per message keeps writes atomic and hydration a trivial SELECT ... ORDER BY seq.
  • Chosen: parts as a JSON array on the message row. Each element is self-describing and mirrors what the bridge emits / useChat expects. SQLAlchemy JSON maps to native JSONB-equivalent on Postgres and TEXT-backed JSON on SQLite. Cost: can't index into individual parts in SQL — which we don't need.

Every part type that must round-trip

Each part object is shaped to match the AI SDK frame the bridge emits (bridge.py::_translate), mapped to its originating frame:

  1. User text — {type: "text", text}. From the inbound POST body, not a frame.
  2. Assistant text — {type: "text", text, state: "done"}. Reconciled from text-start/text-delta/text-end (keyed t_<message_id>).
  3. Reasoning / thinking summary — {type: "reasoning", text, state: "done"}. From reasoning-* (keyed r_<message_id>). What 9dac604 needed; here it round-trips natively.
  4. Tool-call invocation — {type: "tool-<name>", toolCallId, state: "input-available", input}. From tool-input-available (ToolCallEnd).
  5. Tool result (structured_content) — promote the matching tool-call part to state: "output-available" with output. From tool-output-available (ToolExecEnd → _serialize_tool_output, verbatim). The high-fidelity payload the harness transcript loses.
  6. Tool-output error — promote to state: "output-error" with errorText. From tool-output-error.
  7. Stream-level error — {type: "error", errorText} (standalone part; findStreamError in ChatScreen.tsx reads parts[].type === "error"). From the error frame.
  8. Step / finish markers — start, start-step, finish-step, finish are stream-control, not stored as parts. Their semantics live in the status column and message boundaries; agent-ui step markers can be re-synthesized at hydration.

Streaming-vs-final reconciliation

Parts are stored in their final form (state: "done" / output-available). Deltas are accumulated in memory during the turn (see Capture), not persisted per-delta. Ordering within parts follows first-appearance order of each part id across the frame stream — exactly the order the bridge yields them — preserving the visual transcript order.
