Context discovered

The lay of the land, verified against the code. The single most important finding: the harness already persists a transcript, but it is lossy for the UI — and that loss is the whole reason to persist ourselves.

Entry point and bridge

backend/penny/api/main.py exposes POST /api/chat: it builds a per-request Agent bound to a per-chat_id SqliteSession (harness sessions.db in the workspace, separate from penny.db), then streams via stream_agent(agent, prompt). GET /api/sessions/{id} hydrates by reading session.get_messages() through messages_to_ui.

backend/penny/api/bridge.py subscribes to the harness InMemoryEventBus, runs the agent, and maps each Event to one or more AI SDK UI-message-stream frames. This is the only place that sees full fidelity of every part — notably ToolExecEnd.result.structured_content (emitted verbatim via _serialize_tool_output) and the tool-output-error distinction.

The harness already persists a transcript

Inside the loop (agent_harness/core/loop.py), the harness writes to sessions.db incrementally: the user Message at PrepareTurn, each assistant final_message, and each tool_msg. So conversation continuity for the model already survives restarts. The gap is not model memory — it is UI fidelity.

The critical gap: the transcript is lossy for the UI

ToolResultBlock.content is content_to_text(result) — a flat string. The rich structured_content (chart specs, SQL result tables, MCP structured output) the bridge sends to the browser is not stored. Today's messages_to_ui (backend/penny/api/hydration.py) recovers tool output only by json.loads-ing that string — it round-trips by accident when a tool happened to emit JSON-as-text, and silently degrades otherwise. Stream-level Error frames and the tool-output-error vs tool-output-available distinction are also absent from the transcript.

Commit 9dac604 patched one symptom (dropped thinking summaries) by special-casing ThinkingBlock. The plan's stance: stop patching the lossy path; capture the UI-faithful frame stream instead.

Schema reality and frontend

backend/penny/bootstrap.py calls db.create_schema() → Base.metadata.create_all(engine) on startup. No alembic (despite stale references in facade docstrings). New tables are purely additive, so create_all picks them up for free on both SQLite (penny.db) and Neon Postgres.

frontend/src/ChatScreen.tsx: sessionId is a crypto.randomUUID() persisted in localStorage under penny:sessionId; "New chat" just rolls a new UUID and reloads. It already hydrates from GET /api/sessions/{id} before mounting useChat. There is no conversation list and no titling.
