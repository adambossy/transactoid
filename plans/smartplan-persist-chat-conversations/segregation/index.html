Architectural segregation

Penny is two domains that must never commingle. This feature sits squarely in the website domain, and that placement dictates three concrete departures from the naive plan.

Requirements

  1. The finance agent has no way to read or modify stored chat history while doing its job.
  2. Adding conversation storage never makes the agent's code depend on the website's storage code.
  3. A future accidental coupling between the two domains is caught automatically before it ships.

Two domains

  • Agent domain — what the LLM uses to do finance work: backend/penny/tools/ (the @tool wrappers + tools/_services/), backend/.agent/skills/, agent_factory.py, prompts.py, and the prompts. It operates on finance data (derived_transactions, categories, …), notably via the unrestricted run_sql tool.
  • Website domain — the application hosting the agent and owning user-facing CRUD: the FastAPI app (backend/penny/api/), account management, settings, and conversation/message persistence (this feature).

The dependency rule: Website → Agent only

The website constructs and invokes the agent; the bridge is the seam — it runs the agent and translates its event stream. The agent domain must have zero imports of conversation-persistence code, and the persistence code must not import agent tools/skills internals. Conversation persistence is website-owned, not agent infrastructure.

flowchart TD web[Website domain: api/, persistence] --> agent[Agent domain: tools/, skills, agent_factory] agent -. forbidden .-x web bridge[bridge.py: the seam] --- web

Three concrete departures

  1. Do not put conversation models/CRUD in the shared penny/adapters/db facade. That facade and models.py are imported throughout the agent domain (tools/transactions.py, tools/analytics.py's run_sql, tools/_services/{persister,sync_service,split,refund,migrator,…}, the Amazon plugin). Give the website its own persistence package with its own Base, models, and store/engine (backend/penny/api/persistence/{models,store,engine}.py). The shared adapters/db stays finance-only.
  2. Keep conversation data out of run_sql's blast radius. run_sql (tools/analytics.py → get_db().execute_raw_sql) is intentionally unrestricted read+write over the finance DB. Store conversations in a separate database/schema owned by the website's own engine: a dedicated Neon schema (e.g. web.conversations) with run_sql's role/search_path scoped to the finance schema, or a separate SQLite file in dev (penny_web.db) — mirroring the harness's existing "sessions.db is separate" precedent. Defense-in-depth: even with separation, scope run_sql to the finance schema explicitly.
  3. The bridge is the only shared touchpoint — and that's allowed, because it is website code (website → agent). What must not happen: a tool/skill importing the persistence store to "log itself", or the persistence layer reaching into tools/_services for serialization helpers. Those part-shaping helpers stay in the website domain alongside _translate.

The guardrail test

A lint/test guardrail (backend/tests/test_domain_segregation.py) enforces this: assert nothing under penny/tools or the skills tree imports penny.api.persistence, and that the persistence package imports neither penny.tools nor penny.agent_factory. It fails the suite so an accidental import is caught at CI, not in review. See the full watch-list on Cross-dependencies.
