Section 3

Hydration / replay and API surface

Replace the lossy hydration path outright, and ship exactly one persistence-facing endpoint. The lossy reconstruction is deleted, not kept as a fallback.

Requirements

  1. Re-opening a conversation shows it as it streamed, with no special-cased recovery of tool output.
  2. Starting a new chat needs no extra server round-trip — it just works.
  3. A conversation picks up a readable title on its own, with no manual step.

Hydration: replace the lossy path outright

Today GET /api/sessions/{id} reads the harness transcript via messages_to_ui. Switch it to read the new conversation_messages rows, which already hold UIMessage-shaped parts:

  • New conversation_to_ui(rows) -> list[UIMessage] in hydration.py: for each row emit {id: ai_sdk_message_id or f"hist_{seq}", role, parts} — parts pass through nearly verbatim (already UI shape). Because structured_content was stored faithfully, the ThinkingBlock special-case from 9dac604, the _parse_maybe_json heuristic, and _collect_tool_results are all dead — delete them along with messages_to_ui itself.
  • conversation_to_ui is the only hydration path. There is no runtime fallback to the lossy harness transcript — the website never reads sessions.db to reconstruct UI messages. Pre-feature conversations simply do not appear (a clean break, acceptable because persist_session=False means new conversations live only in the app store going forward).
  • If agent-ui needs explicit step markers, re-synthesize start-step/finish-step boundaries at hydration from message/role boundaries rather than persisting them.

API surface: one endpoint kept

  • Hydration (single conversation) — KEPT. Repoint the existing GET /api/sessions/{id} to the faithful path: read conversation_messages rows and return {sessionId, messages: conversation_to_ui(rows)}. Response shape stays {messages}, so the frontend hydration fetch is untouched (zero frontend change). The path name is now a slight misnomer (it reads the conversation store, not the harness session) but keeping it avoids a frontend edit; renaming to /api/conversations/{id} is a later cosmetic.
  • Creation is lazy — no POST /api/conversations. The frontend keeps sending body.id (its localStorage UUID) to POST /api/chat; the backend treats it as conversation_id and calls ensure_conversation(id) on the first chat. No create endpoint, no "New chat" round-trip — "New chat" still just rolls a new local UUID.
  • No manual rename — no PATCH /api/conversations/{id}. Titling is an internal write only: derive a title from the first user message (truncate) on the first assistant finish, written by the persistence layer (no extra model call, no endpoint). A future manual-rename or async LLM-titling job can add the endpoint then.

Deferred (not built now)

  • GET /api/conversations (the conversation LIST) and any preview/updated_at DESC ordering — deferred with the sidebar.
  • Frontend conversation sidebar. No list UI, no server-side "New chat". The frontend is unchanged: it still hydrates one conversation via GET /api/sessions/{id} and mints local UUIDs. The sidebar + list endpoint land together in a later feature.
