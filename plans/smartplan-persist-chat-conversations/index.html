Implementation plan

Persisting Chat Conversations & Messages

Penny streams a rich chat transcript to the browser, but nothing durably stores the UI-faithful record. This plan persists that record where it is born — at the bridge — into a website-owned store kept strictly out of the agent's reach, then replays it back into both the browser and the model.

Requirements

  1. Anyone returning to a past chat sees it re-rendered exactly as it streamed the first time — charts, tables, reasoning, and tool errors intact.
  2. A page refresh in the middle of a reply still shows the in-flight answer, and a dropped connection never loses the question that was asked.
  3. The finance agent can never read or alter a person's stored conversation history as a side effect of answering a money question.
  4. After a restart, the assistant still remembers what was said earlier in the same conversation.

The goal

Today a chat's full fidelity exists only as it streams. The harness writes a transcript to sessions.db for the model, but that transcript is lossy for the UI: tool results are flattened to a string, so chart specs, SQL result tables, and the tool-error distinction are dropped. Hydration on refresh recovers tool output only by accidentally json.loads-ing that string. Commit 9dac604 patched one symptom (vanishing thinking summaries) by special-casing reasoning blocks — a sign the lossy path needs replacing, not patching.

The fix is to capture Penny's own UI-faithful record at the bridge, as frames stream, so that what we persist equals what we streamed by construction.

Strategy

Two new tables — conversations and conversation_messages — live in a website-owned, agent-isolated persistence package on its own SQLAlchemy Base and its own engine, in a separate database/schema from the finance data. A message stores its ordered parts as a JSON array shaped exactly like the AI SDK frames the bridge emits, so hydration is near-passthrough.

We then disable the harness's own session writes (persist_session=False) so the website store becomes the single record — both the UI-canonical history and the source we replay into the model for multi-turn continuity. A dedicated reverse-map module turns stored parts back into harness messages to seed each run.

Headline tradeoffs

  • Capture at the bridge, not from the transcript. Wins on fidelity and live/replay parity; cost is an accumulator that must mirror _translate's id/state conventions — mitigated by shared helpers and a round-trip test.
  • JSON parts column, not a polymorphic parts table. Atomic writes, near-passthrough hydration, no cross-part queries needed; accept losing in-SQL part indexing.
  • Separate website store, not the finance DB. Keeps conversation data out of run_sql's blast radius; cost is a second engine and its own create_all. Segregation here is a hard constraint, not a preference.
  • One store, not two. persist_session=False collapses model memory and UI record into the app store; the reverse-map (a first-class component) is what makes that safe.

How to read this plan

Start with Context discovered for the verified lay of the land, then Architectural segregation — the hard constraint that shapes everything else. Data model, Capture, and Hydration & API are the core write/read paths; Schema & migration covers create_all-only rollout. Model continuity is the load-bearing reverse-map that lets us collapse to one store. Files, risks & sequencing holds the six-commit plan and the segregation watch-list. Assumptions records what is taken for granted.

Scope note: this rollout deliberately trims the conversation LIST endpoint, the frontend sidebar, manual rename, POST/PATCH endpoints, the per-account model, and per-part byte caps — all deferred to later features.
