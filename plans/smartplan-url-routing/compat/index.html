Deployment & compatibility

What already works, what existing users will notice, and how this change dovetails with the auth widgets and the future landing page.

Requirements

  1. A conversation link shared from the deployed product opens correctly for its owner — not a blank page or a server error.
  2. Existing users lose nothing: every past conversation remains reachable from the history drawer.
  3. Nothing in this change has to be undone when the public landing page arrives.

Serving deep links — already solved

Both serving layers already fall back to the app shell for unknown paths, so no deploy-domain change is needed — a happy consequence of the deploy/app segregation:

  • Production — deploy/frontend/nginx.conf.template already contains the SPA fallback: try_files $uri $uri/ /index.html; under a comment that says exactly this ("SPA: fall back to index.html for client-side routes"). /c/<id> serves the shell today; only the client didn't know what to do with it.
  • Dev — Vite's dev server ships SPA fallback by default (appType: "spa"), and the /api proxy matching is prefix-based, so app routes never collide with API calls.

Legacy client state

Existing browsers hold a penny:sessionId local-storage key that pinned their "current" conversation. After this change it is simply never read again — no migration, no cleanup pass (an orphaned key is harmless; deleting it would be code that exists only to delete code's past).

The one user-visible behavior change: returning users land on a new chat instead of their last conversation. Their history is one drawer-click away, and every future visit benefits from the simpler rule. This was an explicit decision (asked and answered), not a side effect.

Clerk widgets and the future landing page

  • Clerk — the sign-in/sign-up widgets use hash routing internally, orthogonal to the path router; their cross-links (/ ↔ /sign-up) are ordinary navigations the router now handles client-side. No Clerk dashboard configuration changes: no redirect URLs move.
  • Landing pages (bead fly-ikm) — that bead depends on this one. The prepared seam: / stays the app's root now; when landing pages arrive, the / route element becomes auth-conditional (signed-out visitors see the landing page, signed-in users see the new chat), and marketing routes join the top-level table beside /ui. Conversation, settings, and invite URLs are untouched by that future change — nothing in this plan gets redone.
