Architecture

Router integration

Where the router sits in the component tree, how it coexists with the two auth modes, and the one keying rule that makes conversation switching correct.

Requirements

  1. The app behaves identically whether sign-in is handled by the hosted auth provider or by the local development stub — routing must not fork into two implementations.
  2. Moving between conversations keeps the app chrome (drawer, header) mounted and stable; only the chat area changes.
  3. The design-system preview remains reachable in development builds and absent from production, exactly as today.

Component tree

One BrowserRouter wraps everything in main.tsx. Beneath it, the existing mode split (Clerk configured or not) is unchanged — but both branches render the same shared route table, differing only in the getToken they inject and the Clerk chrome. This collapses today's duplicated authedScreen() / devScreen() switch into one <AppRoutes getToken={...} /> component — the routing equivalent of the existing pattern where AppShell already serves both modes.

flowchart TD router["BrowserRouter"] --> mode{"Clerk key configured?"} mode -->|"yes"| clerk["ClerkProvider + AuthGate"] mode -->|"no"| dev["Dev-principal mode"] clerk --> shell["AppShell — drawer + header"] dev --> shell shell --> routes["Shared route table"] routes --> chat["Chat screen, keyed by conversation id"] routes --> billing["Providers and billing"] routes --> invites["Invites"]

AppShell becomes a layout that stays mounted across route changes (rendered above the route table, as the diagram shows), so the drawer and header never flicker during navigation. The routed screen renders where children renders today.

Auth modes and the sign-up path

AuthGate keeps its role — signed-out visitors see Clerk's <SignIn>/<SignUp>, signed-in users see the shell — with one fix: its module-level const showSignUp = window.location.pathname.startsWith("/sign-up") becomes a useLocation() read inside the component. Module-level path constants are load-time snapshots; every one of them (three in main.tsx, one in AuthGate.tsx) must die for client-side navigation to be trustworthy.

Clerk's own widgets keep routing="hash", so they manage their internal multi-step state in the URL fragment and never collide with the path router. The signInUrl="/" / signUpUrl="/sign-up" cross-links keep working as ordinary navigations.

A signed-out visitor deep-linking /c/<id> hits AuthGate's signed-out state, signs in, and — because the gate swaps content in place without redirecting — lands on the conversation they asked for. No return-URL plumbing needed.

The keying rule for the chat screen

ChatScreen stops calling loadOrCreateSessionId() and instead receives its conversation id from the route: useParams() on /c/:id, or a freshly minted in-memory UUID on /. The critical rule:

  • Key the screen by conversation id — <ChatScreen key={id} ... />. A different id must remount the screen so hydration re-runs and useChat (which caches by its id option) gets a fresh instance. Without the key, React would reuse the component across drawer clicks and show the previous transcript.
  • The same id must not remount — the / → /c/<id> URL replacement keeps the key stable, preserving the in-flight first-message stream (New-chat lifecycle).

Everything inside ChatScreen — hydration, resume-on-visibility, error surfacing, the session-mode picker — is untouched. Routing changes only where the id comes from.

The dev gallery and other top-level routes

The /ui gallery keeps its two properties: dev-only (import.meta.env.DEV guard — the route is simply not registered in production builds, so it falls into the catch-all redirect) and auth-free (registered above the Clerk/mode split, directly under BrowserRouter). The top level therefore looks like: gallery route (dev only) → everything else into the mode-split shell.

This preserves today's semantics exactly while making the guard a route-registration decision instead of a load-time boolean.
