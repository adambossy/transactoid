URL routing for the Penny frontend

Give every view its own address so conversations can be shared, revisited, and navigated with the browser's own controls.

Requirements

  1. A user can copy the address of a conversation, open it later or on another device, and see the same conversation with its full history.
  2. The browser's back and forward buttons move between conversations and screens the way they do on any normal website.
  3. Each key view — a new chat, a specific conversation, settings, invites — has its own address a user can bookmark.
  4. Switching between conversations feels instant, with no full page reload.

The goal

Today the frontend has no router. Which screen renders is decided by ad-hoc prefix checks on the page path, evaluated once at module load; which conversation is on screen lives in browser local storage; and switching conversations does a full page reload. Nothing about the current conversation appears in the URL, so nothing can be deep-linked, and back/forward do nothing meaningful.

This plan introduces client-side routing so that the URL becomes the single source of truth for what is on screen. The conversation id moves out of local storage and into the path (/c/<id>); the existing settings, invites, and sign-up paths become real routes; and all navigation becomes client-side transitions handled by the router.

Strategy

Adopt React Router v7 in plain declarative mode — routes as components, nothing from its framework tier (no loaders, no code-gen). The route surface is small (~6 routes), so the router stays a thin layer: one BrowserRouter at the top, one shared route table used by both auth modes (Clerk and dev-principal), and links instead of reload helpers.

The work lands in three small, independently verifiable stages: (1) introduce the router with the existing screens mapped at their existing paths — zero behavior change; (2) add the conversation routes and retire the local-storage session mechanism; (3) polish — unknown-path redirect, new end-to-end specs for deep-linking and back/forward, and the requirements-of-record update.

The deployment layer is already ready: the production web server falls back to the app shell for unknown paths, so deep links work the moment the client can route them. See Deployment & compatibility.

Key decisions and tradeoffs

  • / is always a new chat. The previous "reopen the last conversation" behavior (via local storage) is deliberately dropped; history lives in the drawer, and returning users click a conversation to resume it. This trades a small habit change for a much simpler mental model: one URL, one meaning. Details in New-chat lifecycle.
  • Short conversation paths, /c/<id>. Matches the convention users already know from other chat products; leaves room for future top-level routes.
  • The app stays rooted at /. The future public landing page (bead fly-ikm) will make / auth-conditional later; this plan avoids moving every authed URL now. See compatibility.
  • Full client-side navigation. The reload-based switching helpers are retired, not wrapped. The chat screen remounts per conversation id, reusing the existing hydration and stream-resume logic unchanged. See Router integration.
