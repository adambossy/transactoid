Execution

Implementation stages

Three dependency-ordered commits on feature/url-routing, each leaving the app working and the test suite green.

Requirements

  1. At no point during the work is the app broken — each stage can ship on its own if the work pauses.
  2. Existing users notice nothing until the conversation URLs appear; settings and invites keep their current addresses throughout.
  3. When the work is done, the project's requirements-of-record reflect the new deep-linking behavior.

Stage 1 — router shell, zero behavior change

  • Add the react-router dependency (v7, the single consolidated package — not the legacy react-router-dom shim).
  • Wrap the app in BrowserRouter in main.tsx; convert the four module-level pathname constants (/ui, /settings/providers, /invites in main.tsx; /sign-up in AuthGate.tsx) into routes and a useLocation() read.
  • Collapse authedScreen() / devScreen() into one shared AppRoutes used by both auth modes (component tree).
  • Chat still uses the local-storage session id at / — behavior is byte-for-byte today's.

Gate: full existing Playwright suite passes untouched. This proves the router landed without semantic change before any semantics change.

Stage 2 — conversation routes, session module retired

  • Add /c/:id; key the chat screen by conversation id; / mints an ephemeral id and replace-navigates on first send (lifecycle).
  • Convert drawer rows and "New chat" to links; highlight from the route param (details).
  • Delete session.ts; add the catch-all redirect to /; render the "Conversation not found" state on hydration 404 (contract).
  • No existing spec seeds the local-storage key, so none need repair — and this stage is what makes the pre-written conversation-isolation.spec.ts (which already asserts /c/<id> URLs and a visible not-found message) actually passable (testing).

Gate: updated suite passes; manual pass over the edge cases.

Stage 3 — new coverage and the record

  • New Playwright specs: deep-link hydration, back/forward across two conversations, new-chat URL replacement (list).
  • Update REQUIREMENTS.txt (Product section): key views are routable; conversations are deep-linkable; browser navigation works. Per repo rules this lands in the same change that alters the behavior — stage 3 is part of the same PR.
  • Close the loop on the bead: verify each fly-wz0 acceptance criterion against the deployed preview, then close it.

Verification per stage

Frontend-only change; the backend gates don't apply. Each stage runs:

  • npm run build (type-check + production build) from frontend/.
  • npm run e2e (Playwright) — the specs listed on the testing page.
  • A manual dev-loop pass (backend on :8000, Vite dev server) covering drawer switching, refresh mid-conversation, and back/forward — the flows that automated specs approximate but a human should feel at least once.

One PR, three commits, reviewed as a unit; the stage boundaries exist so a reviewer can follow the change and so a bisect lands on a working tree.
