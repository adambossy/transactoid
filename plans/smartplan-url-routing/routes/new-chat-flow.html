New-chat lifecycle

How / becomes /c/<id> — when the id is minted, what lands in browser history, and what the back button means.

Requirements

  1. Opening the app's home address always gives a clean, empty chat, ready for a first message.
  2. The moment a conversation exists, its address appears in the location bar so the user can share or bookmark it immediately.
  3. Pressing back from a conversation never lands the user on a ghost empty chat that duplicates the one they are in.

The lifecycle

On /, the chat screen mounts with a freshly minted UUID held in memory only — nothing persisted, nothing in the URL yet. The history-hydration fetch is skipped (a brand-new id has no history to load; today's code fetches and treats the 404 as empty — we can simply not ask). When the user sends the first message, the existing transport already POSTs with that id and the backend creates the conversation; at that moment the client replaces the URL with /c/<id>.

sequenceDiagram participant U as User participant R as Router participant C as Chat screen participant B as Backend U->>R: visit / R->>C: mount with fresh id, held in memory U->>C: send first message C->>B: POST /api/chat with that id C->>R: replace URL with /c/id B-->>C: stream the reply

The URL swap must not remount the chat. The route element for / and /c/:id renders the same screen keyed by the conversation id, and the id is unchanged across the replace — so React reconciles in place and the in-flight stream is untouched. This is the one subtle implementation constraint on this page; Router integration covers the keying rule.

History semantics: replace, not push

The / → /c/<id> transition uses navigate(..., { replace: true }). It is a rename of the view the user is already looking at, not a navigation — the empty-chat state no longer exists to go back to. With replace, back from a started conversation returns to whatever preceded / (often nothing, or the drawer's previous conversation), which matches what users expect from other chat products.

Every other transition — opening a conversation from the drawer, "New chat", settings, invites — is a normal push, so back/forward walk through them naturally.

Design-it-twice: the rejected alternative. Minting the id at navigation time (the "New chat" button navigates straight to /c/<new-id>) was considered and rejected: it puts URLs for conversations that don't exist yet into history and the location bar, litters the drawer flow with empty conversations if the user wanders off before typing, and makes refresh-on-empty hit the not-found hydration path. Ephemeral-until-first-message avoids all three.

Edge cases

  • Refresh on / before the first message — a new id is minted; the old one never reached the server, so nothing leaks. Any draft text is lost, same as today.
  • Refresh mid-stream on /c/<id> — unchanged from today: hydration sees the transcript ending on a user message and immediately resumes the live stream (the existing initialIncomplete + resumeStream path). Routing does not touch this logic.
  • Switching conversations mid-stream — client-side nav unmounts the streaming chat; the server keeps running the turn (existing behavior when a tab backgrounds). Navigating back to that conversation remounts, hydrates, and resumes the stream. No new code — this is the same resume path.
  • Deep link to a foreign or unknown id — hydration 404s and renders a "Conversation not found" state (message + link to start a new chat) instead of today's silent empty chat. The pre-written isolation spec (e2e/conversation-isolation.spec.ts) already asserts this, and it removes the wrinkle where sending a message into a foreign URL would create a conversation with that id.
  • Session-mode picker — the individual/joint picker shows only on an empty chat, which after this change is exactly the / route. The old 404-as-empty-chat path that also showed it is gone (it is now the not-found state), so the picker's appearance and the "new chat" concept coincide exactly.
