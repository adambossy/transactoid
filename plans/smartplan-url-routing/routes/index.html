Route map & URL contract

Every view the app can show, its address, and who can see it. This table is the contract the rest of the plan implements.

Requirements

  1. A user can bookmark any screen — a conversation, settings, invites — and get back to exactly that screen from the bookmark.
  2. A signed-out visitor who follows a deep link is asked to sign in, and lands on the screen they asked for once signed in.
  3. A mistyped or dead address never strands the user on a blank page — it lands them somewhere useful.

The URL is the source of truth

The invariant this plan establishes: everything needed to reconstruct the current view is in the path. Concretely:

  • The conversation id is read from the route parameter, never from local storage. The penny:sessionId key and the module that owns it are retired (details).
  • Refreshing any URL re-renders the same view: /c/<id> re-hydrates that conversation's history; /settings/providers reopens settings.
  • Back/forward work because every view change is a history entry (with one deliberate exception — the new-chat URL replacement, covered in New-chat lifecycle).
  • Module-level path checks (evaluated once at page load, in main.tsx and AuthGate.tsx) must all become route matches or useLocation() reads — a value captured at load time is stale the moment client-side navigation exists.

Unknown paths

A catch-all route redirects anything unmatched to / (a <Navigate to="/" replace />). With a six-route app there is nothing useful a dedicated 404 page could offer beyond "go home", so we go home directly. Two nuances:

  • The redirect uses replace so the dead URL doesn't linger in history and trap the back button.
  • A well-formed but foreign conversation URL (/c/<someone-else's-id>) is not "unknown" — it matches the route, its hydration 404s, and it renders a "Conversation not found" state with a link to start a new chat. This is a deliberate behavior change from today's silent empty-chat fallback: the pre-written spec e2e/conversation-isolation.spec.ts already asserts a visible not-found message, and it also closes the oddity where messaging into a foreign URL would create a conversation with that id.
